# Joint QDESN Phase 107: Selected VB hardening and freeze plan

Date: 2026-07-07

This note records the audit, diagnosis, and implementation plan for the stage after
Phase 106 VB specification screening. The goal is not to launch MCMC yet. The goal
is to harden the selected VB specification, rerun it under the full synthetic
validation fixtures, and decide whether it can be frozen as the VB initializer for
MCMC.

## GIG Moment Failure Diagnosis

The `rhs_tau0_0p5` candidate failed on `regime_shift` in both fit and forecast with:

`Could not compute deterministic GIG moment.`

The failure is localized to the deterministic GIG moment helper used by exAL VB-LD
scale and latent updates. The original helper used a Bessel-ratio formula with a
numerical integral fallback. Stress testing showed that the fallback itself can fail
for large-order/small-parameter GIG regimes of the form generated by exAL scale
updates. This is a numerical stability defect, not evidence that the synthetic
fixture or artifact pipeline is malformed.

Phase 107 therefore adds a local numerical guard:

- keep the analytic Bessel-ratio path when finite;
- use a stabilized log-kernel integral around the GIG log-mode when the analytic
  ratio is nonfinite or nonpositive;
- clip only final extreme finite moments to machine bounds, preventing worker
  crashes while keeping the event inspectable through convergence, scale, and
  adjustment diagnostics.

## Why This Is The Optimal Next Step

Launching MCMC before this hardening step would mix posterior checking with a known
VB numerical defect. Running another broad VB grid would waste compute because the
dominant signal is already clear: `tau0 = 0.5` with a tighter empirical alpha prior
is the only non-failing, non-catastrophic candidate. The optimal next step is
therefore a selected-spec hardening rerun, not a new exploratory screen.

## Phase 107 Work Plan

1. Preserve Phase 106 evidence.
   - Do not overwrite Phase 106 artifacts.
   - Use a dedicated Phase 107 output directory.

2. Harden deterministic GIG moments.
   - Patch only the GIG numerical fallback.
   - Add large-order regression tests.
   - Keep existing algebra tests for ordinary GIG identities.

3. Rerun the selected specification only.
   - Candidate: `rhs_tau0_0p5_alpha0p5`.
   - Frozen fixture directory: `application/cache/joint_qdesn_simulation_dgp_fixtures_20260706`.
   - Output directory:
     `application/cache/joint_qdesn_selected_vb_freeze_phase107_20260707`.

4. Refresh the Phase 107 audit.
   - Verify top-level and nested manifests.
   - Verify no worker failures, nonfinite values, or contract crossings.
   - Summarize raw crossings, adjustment rates, VB max-iteration rates, and
     fit/forecast truth metrics.

5. Make the freeze decision.
   - `pass`: no implementation failures and review flags are small enough to
     justify freezing.
   - `review`: implementation gates pass but convergence/raw adjustment still need
     narrative or one targeted control rerun.
   - `fail`: any worker failure, nonfinite scores, contract crossing, missing hash,
     or catastrophic scenario row.

## Freeze Criteria Before MCMC

MCMC should start only after:

- worker failures equal zero;
- contract crossings equal zero;
- all quantiles, scores, scale summaries, RHS summaries, and traces are finite;
- no catastrophic scenario forecast MAE appears;
- raw crossings are documented and materially below the baseline failure modes;
- VB max-iteration flags are either reduced or explicitly treated as review-level
  approximation evidence;
- the selected run has a complete SHA-256 artifact manifest.

If these criteria hold, Phase 108 should launch VB-initialized MCMC for the frozen
selected specification, starting with `JOINT QDESN RHS` and selected exAL reference
rows before expanding to every model/scenario combination.

## Phase 107 Decision

This stage resolves the implementation blocker, but it does not justify freezing
every model equally. The evidence supports the following decision:

1. Freeze the VB implementation contract and artifact pipeline.
   The reproducibility, manifest, finiteness, and contract noncrossing gates pass.

2. Treat `JOINT QDESN RHS` as the primary VB article candidate.
   It has the best fit and forecast recovery and almost no raw forecast crossing
   burden.

3. Retain `QDESN RHS`, `JOINT exQDESN RHS`, and `exQDESN RHS` as comparison
   models, but discuss their review flags explicitly.
   The independent models require more monotone adjustment, and the exAL models
   are less accurate under the current VB approximation.

4. Do not run another broad VB grid before MCMC.
   The selected specification is now implementation-clean. A new broad screen
   would mostly remeasure known tradeoffs unless the model class itself changes.

5. Before launching full MCMC, run a small Phase 108 MCMC readiness layer using
   VB initialization and the frozen selected specification.
   Start with `JOINT QDESN RHS`, then selected exAL rows, then expand if the
   chain diagnostics are finite and runtime is acceptable.

## Recommended Next Stage

Phase 108 should implement VB-initialized MCMC readiness over the frozen Phase 107
specification. It should consume the same DGP fixtures and selected candidate
controls, preserve the VB summaries used for initialization, and write a separate
artifact directory rather than overwriting Phase 107.

The first launch should be bounded and diagnostic, not yet the final article MCMC
table:

- scenarios: all nine synthetic DGPs;
- primary model: `JOINT QDESN RHS`;
- reference models: selected `JOINT exQDESN RHS` rows if runtime permits;
- initialization: Phase 107 VB coefficients and scale summaries;
- checks: finite draws, chain summaries, posterior quantile noncrossing contract,
  VB-to-MCMC distance, runtime, and manifest hashes.

If Phase 108 passes implementation gates, the final article-scale MCMC table can
then be launched using the same frozen VB specification and artifact contract.
